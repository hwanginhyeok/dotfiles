# Subagent-Driven Development

> **SSOT: hih-dev**
> 이 스킬은 hih-dev의 STEP 4~4.5(구현 + 2-Phase Subagent Review) 참고 원칙입니다.
> 직접 실행하지 않으며, hih-dev 파이프라인이 실제 워크플로우의 단일 진실 공급원(SSOT)입니다.

## 서브에이전트 모델 선택 전략 (hih-dev에 없는 독자 내용)

작업에 필요한 최소 모델을 사용하여 비용과 속도를 최적화한다.

**기계적 구현 작업** (격리된 함수, 명확한 스펙, 1-2개 파일):
→ 빠르고 저렴한 모델 사용. 계획이 잘 지정된 대부분의 구현 작업은 기계적이다.

**통합 및 판단 작업** (다중 파일 조정, 패턴 매칭, 디버깅):
→ 표준 모델 사용.

**아키텍처, 설계, 리뷰 작업**:
→ 사용 가능한 가장 강력한 모델 사용.

**작업 복잡도 신호:**
- 1-2개 파일, 완전한 스펙 → 저렴한 모델
- 다중 파일, 통합 고려사항 → 표준 모델
- 설계 판단 또는 광범위한 코드베이스 이해 필요 → 가장 강력한 모델

## Implementer Status Handling (hih-dev에 없는 독자 내용)

서브에이전트 구현자는 4가지 상태 중 하나를 보고한다. 각각 적절히 처리:

**DONE:** 스펙 준수 리뷰로 진행.

**DONE_WITH_CONCERNS:** 구현자는 작업을 완료했지만 의심을 표시. 진행 전 우려사항을 읽는다.
- 정확성이나 범위에 대한 우려 → 리뷰 전 해결
- 관찰 수준의 의견(예: "이 파일이 커지고 있음") → 메모하고 리뷰 진행

**NEEDS_CONTEXT:** 구현자에게 제공되지 않은 정보가 필요. 누락된 컨텍스트를 제공하고 재실행.

**BLOCKED:** 구현자가 작업을 완료할 수 없음. 블로커 평가:
1. 컨텍스트 문제 → 추가 컨텍스트 제공 후 동일 모델로 재실행
2. 더 많은 추론이 필요 → 더 강력한 모델로 재실행
3. 작업이 너무 큼 → 더 작은 단위로 분할
4. 계획 자체가 잘못됨 → 사람에게 에스컬레이션

**절대** 에스컬레이션을 무시하거나 변경 없이 동일 모델로 재시도하지 않는다.
구현자가 막혔다고 하면 무언가 변경되어야 한다.

## hih-dev 참조 항목 (본 스킬에서 축소)

다음 내용은 hih-dev에 이미 포함되어 있으므로 hih-dev을 참조:

| 항목 | hih-dev 위치 |
|------|-------------|
| 2-Phase Subagent Review (스펙 준수 + 코드 품질) | STEP 4.5 |
| 공유 리소스 보호 (DECISIONS.md, PATTERNS.md 등) | STEP 4.5 내 공유 리소스 보호 섹션 |
| 파일 쓰기 검증 (patch/write_file 후 read_file 확인) | STEP 4.5 내 파일 쓰기 검증 섹션 |
| 병렬 구현 시 리뷰 흐름 | STEP 4.5 내 병렬 구현 시 리뷰 흐름 섹션 |
| 수정 → 재리뷰 루프 | STEP 4.5 내 수정 → 재리뷰 루프 섹션 |
| 전체 파이프라인 (task check → 구현 → 리뷰 → 배포) | STEP 0 ~ STEP 8 |
